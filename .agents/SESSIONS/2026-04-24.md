### Decisions

- **Decision:** Keep kanban under `boards`, not under `flows`
  - **Context:** Kanban represents board/state surfaces, not node-edge editors
  - **Rationale:** `boards` and `flows` are different product primitives and should stay separate

- **Decision:** Use `flows` as the public node-edge surface and keep `workflows` as a compatibility alias
  - **Context:** `workflows` was too broad and blurry for React Flow primitives
  - **Rationale:** `flows` is the cleaner long-term API, but consumers need a migration buffer

- **Decision:** Ship `flows.css` as a separate asset
  - **Context:** XYFlow consumers need base library CSS plus shared theme overrides
  - **Rationale:** A dedicated CSS entry keeps flow-specific styles out of the generic root stylesheet

- **Decision:** Externalize `@xyflow/react` and `recharts`
  - **Context:** Initial builds bundled these heavy runtimes into the library output
  - **Rationale:** Keeping them as package dependencies prevents duplicated heavyweight artifacts in consumer bundles

### Mistakes and fixes

- **Mistake:** Initial flow/chart build inlined `@xyflow/react` and `recharts`, producing oversized subpath bundles
- **Fix:** Marked both libraries as externals in `vite.config.ts`
- **Prevention:** Treat heavy visualization runtimes as externals by default unless there is a specific reason to inline them

- **Mistake:** Initial XYFlow typing used the wrong controls prop type and loose story typing
- **Fix:** Switched to `ControlProps` and explicit `Node<...>` typing for the flow story
- **Prevention:** Validate third-party surface types against the installed library version before wiring public exports
